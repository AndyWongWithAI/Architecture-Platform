## 1. 问题与目标

**问题**:SDLC 8 阶段(requirement → design → code → ci → test → deploy → operate → feedback)在 `~/.claude/sdlc-state/<req-id>/<stage>.json` 写完 manifest 后,arch platform 的 `requirement.status` 字段不会自动推进。当前 33 条需求中,真实已完工的 5 条长期停在 `draft/triaged`,看板无法区分 in-flight vs done。

**目标**:每个 stage 完成后自动调 `arch requirement update --status <mapped>` 推进状态。

## 2. 设计决策

### 2.1 抽象成独立组件 `stage-status-syncer`(L1_platform)

**反对方案**:在 8 个 stage skill 里 inline 写 `arch requirement update` 调用。

**理由**(用户拍板 + 复用原则):
- 解耦:stage 业务逻辑 vs 看板同步,改映射表不用动 stage skill
- 可复用:任何未来想接入 SDLC 状态同步的项目直接调
- 可测试:sync_stage() 单元测试,与 stage 业务解耦
- 失败处理统一:warn-only + 留痕 `status-sync-fail.json` 一处实现

### 2.3 失败处理(warn-only,用户拍板)

```python
def sync_stage(req_id: str, stage: str) -> None:
    try:
        target = STAGE_STATUS_MAP[stage]
        subprocess.run(["arch", "requirement", "update", req_id, "--status", target], check=True)
    except Exception as e:
        logger.warning(f"[stage-status-syncer] failed to sync {req_id} → {stage}/{target}: {e}")
        _write_failure_manifest(req_id, stage, target, str(e))  # 留痕 sdlc-state/<req>/status-sync-fail.json
```

**关键不变量**:`sync_stage()` 永远不抛异常,永不影响 stage 主流程返回。

## 4. 风险与边界

| 风险 | 缓解 |
|------|------|
| 状态机乱序(已是 verified 又收到 in_progress 请求) | arch 后端状态机校验已拦截(实施时实测:triaged → in_progress 被拒) |
| arch CLI 故障导致 SDLC 全停 | warn-only 不阻塞(本设计核心决策) |
| GH Action secret 没配(ARCH_PLATFORM_API_KEY) | 失败留 warning,人工排查 |
| 重复 sync(同一 stage 被多次触发) | 当前 arch PATCH 是 idempotent,但 status 回退不会发生(状态机阻止) |
