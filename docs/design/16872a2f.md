## 4. ⚠️ 关键设计决策:分层口子

`composed_of` 里**有两个潜在分层违规**,需要拍板:

### 4.1 sdlc-workflow 编排其他 L2 skill

sdlc-workflow 是 L2,但要依赖 6 个 L2 skill(sdlc-requirement/design/code/operate/feedback/registry-validate/doubt-driven-development)。严格分层 `L_n → L_{n-1}` 下不合规。

**用户拍板**(2026-06-22 AskUserQuestion):**允许同层依赖**,理由:编排型 skill(sdlc-workflow)本质是 L2 → L2 调度,应在分层原则里开个口子。

**CLAUDE.md 补充注脚**:
```
composed_of(代码逻辑组合):必须逐层 L_n 只能引用 L_{n-1}
  例外(2026-06-22):编排型 skill(sdlc-workflow / sdlc-workflow 类)允许 L2→L2 composed_of。
  LayeringCheck 仅检查非编排型 skill 的跨层调用。
```

### 4.2 sdlc-design 依赖 sdlc-requirement(L2 → L2 同层)

类似情况,但更细:Phase 2 必须读 Phase 1 的 manifest。用户拍板:同上,允许。

## 5. 🚨 阻塞:后端 API 缺口(FB-38f2024f)

**实施时实测发现**:`ComponentUpdate` Pydantic schema 不含 `composed_of` 字段。

```
ComponentUpdate 字段(实测):
  title, category, status, tags, repo_url, language, package_name,
  install_command, usage_example, is_asset, distribution_form,
  interface_contract, knowledge_artifact
  ❌ composed_of(缺!)
```

FastAPI strict 模式(Pydantic v2 默认)会拒收 unknown 字段,返回 422。

**影响**:Phase 2 没法直接 PATCH 现有 13 个 skill 补 composed_of。能做的:
1. ✅ ComponentCreate 时填 composed_of(已验证可用,stage-status-syncer 即是)
2. ❌ ComponentUpdate 不能改 composed_of

**Phase 3 前置**(FB-38f2024f,high):
- 后端 `backend/app/schemas/component.py` `ComponentUpdate` 加 `composed_of: list[ComposedOfEntry] | None = None`
- 同步更新 OpenAPI schema
- 加单元测试:PUT composed_of 后 GET 能读到
